## Narrative

1. **The key lands in every dump.** All four flavours captured the Go
   runtime heap arena at `0xc000000000`, which is where gocryptfs's
   `aes.cipher` struct lives. The expanded-key material appears twice
   (at `0xc0000bc220` and `0xc0000bc4a0`, a 0x280-byte stride) — this is
   consistent with Go's `crypto/aes` keeping a primary `enc` round-key
   table plus a `dec` table in adjacent fields of the same struct.

2. **ASLR base is unrelated to the key.** `aslr_base = 0x400000` is the
   program text load address, not the heap. The key sits on the
   Go-runtime heap at `0xc0000bc220`, which is the normal arena start on
   `linux/amd64` (`0xc000000000 + offset`). This is a documentation
   note — no pipeline bug.

3. **All four `find_all(..., view=...)` implementations agree.** The
   number of hits matches between raw and vas views for every flavour.
   `MslDumpSource`, `GdbRawDumpSource`, `LldbRawDumpSource`, and
   `GCoreDumpSource` each expose `find_all(needle, view)` with
   equivalent semantics — no gap found, no fallback needed.

4. **Strings extraction wall-clocks are flat ≈ 0.38 s** for the first
   500-result page across all four dumps at `min_length=6`, including
   the 2.0 GB gcore. Chunked streaming (8 MiB) is working as designed.

5. **Pipeline status:** end-to-end headless works on the real dataset.
   The same 32-byte needle is located consistently at the same VA in
   four independently-captured flavours, and the MCP-facing entry points
   (`extract_strings_tool`, `get_session_info`) return correctly-shaped
   responses in under a second on MSL and under four seconds on the 2 GB
   gcore.

## Gaps & follow-ups

- **No source-class gaps.** All four DumpSource classes already expose
  `find_all(needle, view)` with identical semantics. No shims were
  added.
- **MSL raw-view inverse-VA is best-effort.** Because .msl blocks are
  compressed, a raw-file hit cannot be byte-exactly translated to a VA;
  we report the containing region's base as an anchor instead. The vas
  view supports byte-exact inverse translation via `iter_ranges`.
